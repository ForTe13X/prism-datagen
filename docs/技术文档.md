## 1. 总体架构与模块图

整个包分为**核心库 `datagen/`**（纯 stdlib）与**可选界面**（CLI 一样纯 stdlib、Web UI 需 `fastapi`/`uvicorn`）。

```
                            datagen/specs/*.json      (数据域 spec：vocab / roles / params / tasks / notes)
                                     │  load_source / list_sources
                                     ▼
  core.py  ──────────►  generator.py  ──────────►  oracle.py
  (_unit / _wiggle       (generate()、真值先建、        (observation_view +
   确定性原语)            两个旋钮、to_sqlite)           naive / linked / oracle
       ▲                        │                       + score + evaluate)
       │ 仅被 generator          │  package(dict)
       │ 直接依赖               ▼
       │                 materialize.py
       └────────────────  (to_json / to_csv / to_sqlite* / preview)
                                 │
                 ┌───────────────┴────────────────┐
                 ▼                                 ▼
             cli.py                            server.py + web/index.html
        (list / gen / sweep 子命令)        (FastAPI /api/* + 单页 UI，只读)
```

依赖方向（严格单向，避免环）：

| 模块 | 职责 | 依赖 |
| --- | --- | --- |
| `datagen/core.py` | 确定性原语 `_unit`（sha256→[0,1]）、`_wiggle`（平滑信号）。**不 import 任何本包模块。** | 仅 `hashlib`、`math` |
| `datagen/generator.py` | `generate()` 主入口：解析 spec → 真值先建 `_build_truth` → `_throughput` → `_news`（受 link 影响）→ `_apply_dirtiness`（受 dirtiness 影响）→ 组装 package；`to_sqlite`；`load_source`/`list_sources`。 | **只依赖 `core`**（`_unit`/`_wiggle`）+ stdlib |
| `datagen/oracle.py` | `observation_view` 剥出观测视图 → `naive_solve`/`linked_solve`/`oracle_solve` 三解法 → `score`（P/R/F1）→ `evaluate`（一次算 naive/linked/oracle）。 | **依赖 `generator`**（复用 `_DEFAULT_ROLES`）+ stdlib |
| `datagen/materialize.py` | `to_json` / `to_csv` / `to_sqlite`（从 generator re-export）/ `preview`（人读文本）。 | 依赖 `generator`（re-export `to_sqlite`）+ stdlib |
| `datagen/cli.py` / `__main__.py` | `list` / `gen` / `sweep` 三个子命令；`gen --out` 写文件、`--eval` 打印判别力；`sweep --over` 扫旋钮。 | 依赖 `datagen` 包公开 API + `argparse` |
| `server.py` + `web/index.html` | 本地只读 Web UI：`/api/specs`、`/api/generate`、`/api/sweep`、`/`；单页 HTML 渲染真值链/污染/吞吐/新闻/运单，并做悬停跨源联动高亮。 | `fastapi`、`uvicorn` + `datagen` |

关键设计约束（源码可验证）：

- **`generator` 只依赖 `core`**：真值与观测的生成完全建立在两个确定性原语之上，不引入任何随机性/时钟。
- **`oracle` 依赖 `generator`**：为了让三解法保持 domain-generic，`oracle.py` 从 `generator` 导入 `_DEFAULT_ROLES` 作为 `roles` 缺省。
- **CLI 与 Web UI 共用同一条生成路径**：`server.py` 的注释明确「Everything the UI shows comes from the SAME deterministic generator the CLI uses — no separate code path.」两个界面都只是调用 `generate()`/`evaluate()`。

## 2. 确定性原理

**载荷性质：逐字节可复现（byte-reproducibility）。** 相同的 `(source_id, dirtiness, link_explicitness, seed)` 组合，在任何机器上都产出**逐字节相同**的 package。测试 `tests/test_determinism.py::test_same_knobs_byte_identical` 用 `json.dumps(..., sort_keys=True)` 规范化后断言两次生成完全相等。

实现的根基是 `core.py` 的两个函数，**无 `random`、无时钟、无环境态**。

### 2.1 `_unit(*parts) -> float`：sha256 → [0,1]

```python
def _unit(*parts: object) -> float:
    seed = "|".join(str(p) for p in parts)
    return int(hashlib.sha256(seed.encode("utf-8")).hexdigest()[:8], 16) / 0xFFFFFFFF
```

- 把任意个「种子片段」用 `|` 连成字符串，取 sha256 十六进制摘要**前 8 位**（32 bit），除以 `0xFFFFFFFF` 归一到 `[0,1]`。
- `ffffffff` 恰好映射到 `1.0`（分母就是 `0xFFFFFFFF`）。
- 因为是纯哈希，同样的 parts 必得同样的浮点数 —— 这是所有下游"随机感"的唯一来源。

`generator.py` 在此之上包了两个便捷函数：`_u(seed, *parts) = _unit(seed, *parts)`，以及取整数区间的 `_ri(seed, lo, hi, *parts)`（`lo + int(_u(...) * (hi-lo+1)) % (hi-lo+1)`，当 `hi <= lo` 直接返回 `lo`）。

### 2.2 `_wiggle(frame, *seed) -> float`：平滑确定性信号

```python
def _wiggle(frame: int, *seed: object) -> float:
    f1 = 0.03 + 0.09 * _unit(*seed, "wf1")   # 慢趋势，周期约 11–33 帧
    p1 = _unit(*seed, "wp1")
    f2 = 0.13 + 0.17 * _unit(*seed, "wf2")   # 更快的小涟漪
    p2 = _unit(*seed, "wp2")
    main   = math.sin(2*math.pi*(f1*frame + p1))
    ripple = 0.3 * math.sin(2*math.pi*(f2*frame + p2))
    return (main + ripple) / 1.3
```

- 返回一个在 `~[-1, 1]`、均值 ≈ 0 的**平滑信号**：一条慢趋势 + 一个 0.3 幅度的更快涟漪。
- 频率/相位都由 `_unit(*seed, ...)` 哈希导出，因此对同一 `seed` 是固定的正弦曲线；对 `frame` 连续，所以吞吐量时序看起来"平滑漂移"而非白噪声。
- 用途：`_throughput` 用它把每个仓的吞吐量在基线上下漂移（`v = base + amp * _wiggle(t, seed, "tw", w["id"])`）。

### 2.3 种子构成

`generate()` 里 `sd = str(seed) if seed else str(src.get("seed", source_id))` —— 显式 `seed` 优先，否则用 spec 的 `seed` 字段（`logistics_demo.json` 里是 `"logistics_demo"`），再退化到 `source_id`。

之后每一处"随机"都把 `sd` 作为第一个片段，再拼上**语义标签 + 索引**，例如：

- 运单落哪个仓：`_ri(seed, 0, nw-1, "shipwh", j)`
- 单位漂移触发：`_u(seed, "dirtylb", s["id"])`
- 吞吐涟漪：`_wiggle(t, seed, "tw", w["id"])`

**为何逐字节可复现**：每个值都是 `sha256(seed | 语义标签 | 索引)` 的纯函数；没有全局计数器、没有 `time`、没有 `random` 的隐藏状态。改 `seed` 会整体改变输出（`test_seed_changes_output_but_stays_reproducible` 断言 `ho-0 ≠ ho-1` 但各自可复现）。语义标签避免了不同用途"撞种子"—— 例如 `"shipwh"` 与 `"shipcr"` 即便索引相同也走不同哈希桶。

## 3. 数据模型与不变量：真值先建、观测后附

### 3.1 因果真值链

预埋的跨源因果链是：**新闻事件 → 吞吐量异常 → 延误运单**。

`_build_truth` 是唯一"作者"真值的地方：

1. 先按 spec 的 counts 造出 `warehouses` / `carriers` / `shipments`（运单初始状态 `in_transit`）。
2. 造 `ground_truth_events` 个事件，**每个事件落在一个不同的 hub 上**（`min(n_gt, nw)`，并用 `used_idx` 保证 distinct）。distinct 是关键：一 hub 一事件，后面按 hub 建的"异常帧映射 / 答案映射"才不会把两条真值折叠成一条（`test_events_on_distinct_hubs`）。
3. 事件帧 `f_e` 锚定到该 hub 在锚定带 `[lo_b, hi_b]` 内的**真实运单**的 `dispatch_frame` 上（若有），保证异常确实作用到某些记录；再夹在 `[2, frames-3]`。
4. 该 hub 上 `abs(dispatch_frame - f_e) <= window` 的运单被标记为 `delayed`（`affected`）；其 id 排序后存入事件的 `shipment_ids`。
5. 其余未受影响的运单确定性地"结算"为 `on_time` 或 `in_transit`（阈值 `settle_threshold=0.3`）。

`_throughput` 再让**每个有事件的 hub** 在其事件帧附近（`af-1 <= t <= af+2`）刻出一个凹陷（`v -= (hi-lo)*(0.45 - 0.1*abs(t-af))`）—— 即"吞吐量异常"。这一步让时序与真值一致。

### 3.2 观测后附

真值建好之后，观测层才被生成/污染：

- `_news`：把每个真值事件变成一条新闻（曝光程度由 link 决定），其余补足为 distractor 干扰新闻。
- `_apply_dirtiness`：在观测上原地注入 6 类污染。

### 3.3 核心不变量：真值对两旋钮不变

**两个旋钮只动观测，永不动真值。** 源码与测试共同保证：

- `_apply_dirtiness` 的 docstring 明确「The ground-truth (event answers) is never touched here.」它只改 `news` 的 body/frame、`shipments` 的 measure/unit/status、`throughput` 的个别点。
- `answers` 由 `truth["events"]` 计算，与 dirtiness/link 无关。
- 测试 `test_truth_invariant_to_dirtiness_and_link`：对 `d ∈ {0,0.3,0.6,0.9} × l ∈ {1..5}` 的全组合，断言 `pkg["ground_truth"]["answers"]` 恒等于基线。
- 测试 `test_dirtiness_never_touches_ground_truth`：`dirtiness=0.0` 与 `0.9` 的 `ground_truth` 全等。

另一条不变量：**答案用观测可见的 id 作键**。`answers.explain_delays` 的键是**新闻 id**（`NEWS-00x`），不是内部事件 id `EVT-x`；`anomaly_cause` 的键是 hub id。这样只看 stores 的解法也能被打分（`test_answers_keyed_by_observable_news_ids`）。内部标签 `_truth_event` 只在生成期用来把事件连回新闻，`observation_view` 会剥掉它（`test_observation_view_hides_truth_metadata`）。

## 4. 两个旋钮的机制细节

### 4.1 `dirtiness` 0–1：6 类污染 + corruption_map

`_apply_dirtiness(seed, d, warehouses, shipments, throughput, news, cfg)` 按强度 `d` 原地污染观测，并把每处污染记入可逆的 `corruption_map`（variant→canonical）。`d <= 0` 时直接返回空 map（`test_zero_dirtiness_is_clean`）。

6 类污染各自的**触发概率**（每个候选对象独立用 `_u(seed, 标签, id) < 阈值` 判定，阈值随 `d` 线性缩放）：

| 类别 | 语义 | 触发条件（阈值） | 记入 corruption_map |
| --- | --- | --- | --- |
| identity（别名改写） | 把新闻正文里的 region token 换成别名（如 `华东→华東/HD`），制造实体消歧压力 | `_u(...,"dirtyalias",n.id) < d` | `aliases: {variant: region}` |
| unit（单位漂移 kg→lb） | 部分运单 measure 存成 lb（`×2.20462`），`unit="lb"` | `_u(...,"dirtylb",s.id) < d*0.5` | `weight_lb_ids: [id,...]` |
| missing（状态置空） | 把部分运单 status 置 `None`，打击"读 affected 状态"的解法 | `_u(...,"dirtynull",s.id) < d*0.3` | `status_nulled_ids: [id,...]` |
| time（时间偏移） | 把部分新闻 frame 平移 `±1..2`，打击时空对齐 | `_u(...,"dirtytime",n.id) < d*0.4` | `news_time_offset: {id: offset}` |
| numeric（数值冻结） | 把个别吞吐点冻结成前一帧值（传感器故障示意） | `_u(...,"dirtyfreeze",wid,t) < d*0.05` | *（不单列，直接改序列）* |
| encoding（编码乱码） | 把部分新闻 body 做 `utf-8→latin-1` 乱码（GBK↔UTF mojibake 示意） | `_u(...,"dirtygarble",n.id) < d*0.25` | `garbled_news: [id,...]` |

可逆性由测试保证：`test_unit_drift_recorded_and_reversible` 断言"存成 lb 的运单集合 = corruption_map 里记录的集合"，因此打分器可以按 map 换算回 kg。`test_corruption_scales_with_dirtiness` 断言污染总量随 `d` 单调增（`d=0.9 > d=0.3 > 0`）。

`corruption_map` 固定结构（即使某类为空也保留键）：

```python
{"aliases": {}, "weight_lb_ids": [], "status_nulled_ids": [],
 "news_time_offset": {}, "garbled_news": []}
```

> 注意：numeric 冻结不在 map 里单列，它直接改写吞吐序列，不产生 variant→canonical 记录。

### 4.2 `link_explicitness` 1–5：L1–L5 新闻模板

`_news` 里 `level = 1 if link<=1 else (5 if link>=5 else link)`，取 `vocab.news[f"l{level}"]` 模板格式化。曝光度从"直接写 id"到"纯语义"逐级降低（`logistics_demo` 的模板）：

| Level | 模板（vocab.news） | 暴露了什么 |
| --- | --- | --- |
| L1 | `{kind}影响 {hub_id}({hub_name});受影响{record_label}:{ids}。` | **字面运单 id + hub id + hub 名** |
| L2 | `{kind}波及『{region}{name_suffix}』一带,多批货受阻。` | region + 名称后缀（`中心仓`） |
| L3 | `{kind}袭{region},约第 {frame} 帧前后港区作业受限。` | region + 帧号 |
| L4 | `第 {frame} 帧前后,某港口因{kind}临时停摆,吞吐骤降。` | 仅帧号 + 事件类型（时空线索） |
| L5 | `{kind}逼近{port}所在沿海,航运预计中断数日。` | 仅 port（纯语义/地理线索） |

distractor 新闻用 `distractor` 模板，`_truth_event = None`，不产生真值效果。所有新闻按 frame 排序。

### 4.3 旋钮夹取

`generate()` 入口即 `d = clamp(dirtiness, 0, 1)`、`link = clamp(link_explicitness, 1, 5)`（`test_knobs_are_clamped`：`-5→0.0`、`99→1.0`、`link -9→1`、`99→5`）。

## 5. 判别力三解法（naive / linked / oracle）

`oracle.py` 在**观测视图**上跑三个解法，目的是证明"任务确实需要跨源推理"。当前只针对 `explain_delays` 任务实现字面/联结解法。

### 5.1 observation_view

`observation_view(package)` 返回 solver 唯一能看的东西：`sql` / `timeseries` / 剥掉 `_` 前缀键的 `news` / `roles`。**不含** `ground_truth`，也不含真值侧的 `window`（`_DELAY_TOL=7` 是解法自带的领域先验，不从真值读，避免生成参数经观测通道泄露）。

### 5.2 oracle_solve —— 天花板

直接返回 `ground_truth.answers[task_id]`。因为它就是真值，`score` 恒为 `1.0`（`test_oracle_is_perfect` / `evaluate` 里 `oracle_f1` 写死 `1.0`）。

### 5.3 naive_solve —— 字面精确匹配、单源

只做字面匹配、不跨源：先收集 status==`delayed` 的运单，按 hub 分组；对每条新闻，先找**正文里字面出现的运单 id**，找不到再找**字面出现的 hub id**。只有 L1 会把 id/hub 写进正文，所以 L1 下 naive 满分、L≥2 崩到 0（`test_naive_perfect_when_link_is_literal` / `test_naive_collapses_when_link_hidden`）。

### 5.4 linked_solve —— 跨源时空 + 实体联结

不需要显式 key。`_anomaly_frames` 先从吞吐序列检出每个 hub 的凹陷帧（`series[lo_i] < mean*0.7` 才算真凹陷）。对每条新闻，对每个 hub 打分（多线索取 OR）：

| 线索 | 加分 |
| --- | --- |
| 正文含 hub id 或 hub 名 | +3 |
| 正文含 region | +2 |
| 正文含 port | +2 |
| 该 hub 有异常且 `|异常帧 − 新闻帧| <= 3` | +2 |

取得分最高的 hub（`best_score>0`），再把该 hub 上 `|dispatch_frame − 新闻帧| <= _DELAY_TOL(7)` 的 delayed 运单作为预测。因为 L2–L5 每级都至少留一条线索，linked 在 L1 之外仍能部分恢复。

**这个骨架的诚实上限**（`oracle.py` docstring 与 `sweep --over link` 观测一致）：确定性 linked **只区分 L1（显式 key）vs L≥2（非显式）**，L2–L5 得分相同（约 0.8）。它是**未来 LLM/axiom 语义解法的确定性占位替身**，不是真正的语义求解器；满分上限 ~0.8 而非 1.0。

### 5.5 score / evaluate

`score(pred, truth)` 把答案 `_pairs` 展平成可比元组集合（dict 值比较 value 不只比字段名），算 precision/recall/F1。`evaluate(package, task_id="explain_delays")` 一次给出 `naive_f1` / `linked_f1` / `oracle_f1(=1.0)` 与 `gap = linked_f1 − naive_f1`。

**真实 sweep 输出（可原样引用）：**

```
sweep --over link:      L1:naive1.000 linked0.800 gap-0.200; L2..L5:naive0.000 linked0.800 gap+0.800
sweep --over dirtiness: d0.0:linked0.800; d0.3:linked0.571; d0.6:linked0.500; d0.9:linked0.500(naive 恒 0)
```

其中 L1 的 `gap = -0.200` 说明：link 显式时，字面解法反而略胜跨源骨架（linked 上限 0.8）；一旦 link 隐晦（L≥2）naive 崩到 0、linked 仍 0.8，gap 转正 —— 这就是**判别区间**。dirtiness 升高时 linked 单调下降是**跨 8 个 seed 的均值**趋势（`test_dirtiness_degrades_linked_on_the_mean` 明确注释：并非逐 seed 单调，单 seed 上污染偶尔把边界匹配"歪打正着"）。

## 6. 可扩展性：spec 驱动，换域零代码

跨源因果 pattern（source-event → metric anomaly → affected records）是**可复用引擎**；一切 domain 特有的东西都放在 spec 的 `vocab` + `roles` 里，因此**换域 = 换 spec，零生成器代码**。

- `vocab`：实体/字段名、id 前缀与宽度、carrier 名字表、measure 字段/量纲/范围、metric 名/范围、status 词表、event kinds、别名表、L1–L5 新闻模板。`_cfg` 用 `{**_DEFAULT_VOCAB, **spec.vocab}` 合并，缺省即 logistics。
- `roles`：把 pattern 的槽位映射到该域的 store/field 名（`hub_store`/`record_store`/`metric_store`/`record_hub_fk`/`record_frame`/`record_status`/`record_ids_field`/`affected_status` 等）。下游判别力/解法/materialize 全部**按 role 读**，不硬编码"warehouses/shipments"。
- `params`：counts（接受泛化 `hubs/agents/records` 或 logistics 命名 `warehouses/carriers/shipments`）、`frames`、`news_events`、`ground_truth_events`、`delay_window`。
- `tasks`：任务清单（`explain_delays`、`anomaly_cause`）随 package 携带。
- `regions` / `ports`：地理词表。
- `notes.scope` / `notes.honesty`：进入 manifest，作为诚实边界随包发布。

`load_source` 会校验 `source_id` 只含字母数字下划线、文件存在且内部 `id` 与文件名一致；`list_sources` 扫 `datagen/specs/*.json`。`to_sqlite`/`to_csv`/`preview`/三解法都是 role-generic，所以任意域的 package 都能落地和评测，而非只有 logistics。

## 7. 数据契约：package 结构

`generate()` 返回的 `package`（dict）契约如下（键固定）：

```python
{
  "source_id": str, "seed": str, "dirtiness": float, "link_explicitness": int,
  "roles": {...},                       # 让 generic 解法/落地找到各域 store/field 名
  "stores": {                           # 观测层
    "sql": {                            # 真正关系型：一 store 一 table
      "warehouses": [ {id, region, port, name}, ... ],
      "carriers":   [ {id, name}, ... ],
      "shipments":  [ {id, warehouse_id, carrier_id, dispatch_frame,
                       weight_kg, status, unit}, ... ],
    },
    "timeseries": { "throughput": {WH-id: [float, ...]}, "frames": int },
    "news": [ {id, frame, kind, headline, body, _truth_event}, ... ],  # _truth_event 内部
  },
  "ground_truth": {                     # 两旋钮永不改动
    "events":  [ {id, warehouse_id, region, port, frame, kind, shipment_ids}, ... ],
    "answers": {
      "explain_delays": { NEWS-id: [SHP-id, ...] },        # 键为观测可见的新闻 id
      "anomaly_cause":  { WH-id: {frame, news} },
    },
    "window": int,
  },
  "corruption_map": { "aliases": {}, "weight_lb_ids": [], "status_nulled_ids": [],
                      "news_time_offset": {}, "garbled_news": [] },   # variant→canonical，可逆
  "tasks": [ ...spec.tasks... ],
  "manifest": {                         # 摘要 + 诚实边界
    "source_id", "title", "seed", "dirtiness", "link_explicitness",
    "counts": { warehouses, carriers, shipments, news, frames, ground_truth_events },
    "scope": str, "honesty": str,       # 取自 spec.notes
  },
}
```

**落地模态**（`materialize.py`，均 role-generic）：

- `to_json(pkg, path)`：全量 UTF-8 JSON（`ensure_ascii=False`）。
- `to_csv(pkg, outdir)`：每个 SQL store 一个 CSV（`utf-8-sig` 便于 Excel 读 CJK）+ 长表 `throughput.csv`（`hub_id, frame, value`）+ `news.csv`（剥 `_` 前缀键）。
- `to_sqlite(pkg, path)`：把 SQL store 落成**真实可查询**的 SQLite —— 一 store 一 table，列由该 store 首行 schema 推断，`id` 作 PK。`test_sqlite_is_queryable` 证明 `SELECT COUNT(*) ... WHERE status='delayed'` 与包内计数一致。
- `preview(pkg, sample=3)`：确定性人读文本（manifest、真值链、污染清单、各 store 样本）。

`server.py` 的 `/api/generate` 在返回 package 上额外挂一个 `pkg["evaluation"] = evaluate(pkg)` 供 UI 画判别力柱。

**CLI 产物命名**（`gen -o <dir> --format all`）：`<stem>.json` / `<stem>_csv/{warehouses,carriers,shipments,throughput,news}.csv` / `<stem>.db`，其中 `stem = <domain>_d<dirtiness>_l<link>_<seed>`。

## 8. 性能与依赖

**依赖分层**（`requirements.txt` / `pyproject.toml` 明确）：

- **核心生成器 + CLI：纯 stdlib、零第三方依赖**。用到 `hashlib` / `json` / `csv` / `sqlite3` / `argparse` / `math` / `pathlib`。`pyproject.toml` 的 `dependencies = []`。
- **可选 Web UI**：`fastapi>=0.110`、`uvicorn>=0.29`（`[project.optional-dependencies].web`）。只有跑 `server.py` 才需要。
- **开发/测试**：`pytest>=8.0`（`[...].dev`）。
- `requires-python >= 3.10`。

**性能特征**（源码可推断）：

- 生成规模由 spec 的 counts 决定（`logistics_demo`：3 仓 / 4 承运 / 24 运单 / 60 帧 / 8 新闻 / 2 真值事件），全部是内存中的小 list/dict + O(记录数) 的循环，无 I/O、无网络、无锁。生成一个包是毫秒级、常数内存。
- `_throughput` 对每仓每帧调一次 `_wiggle`（内含 4 次 `_unit`/sha256）；`_apply_dirtiness` 对每个候选对象各一次 `_unit`。计算量与 `hubs × frames` + 记录数线性相关。
- `sweep` 就是对 4 个（dirtiness）或 5 个（link）取值各生成+评测一次，串行、无缓存，但每次都极轻量。
- 确定性带来一个免费好处：**可缓存 / 可 diff / 可回归**——同参数产物逐字节稳定，适合做基准与快照测试。

**入口方式**：

```bash
python -m datagen list                                   # 列数据域
python -m datagen gen -d 0.6 -l 4 -s ho-0 --eval         # 预览 + 判别力
python -m datagen gen -o examples --format all           # 落地 json/csv/sqlite
python -m datagen sweep --over link                      # 扫旋钮曲线
pip install fastapi uvicorn && python server.py          # 本地 UI（127.0.0.1:8123）
```

CLI 会先 `_fix_console()` 把 stdout/stderr 重配为 UTF-8，避免 Windows GBK 控制台在 CJK 预览上崩溃。

## 优缺点（pros & cons）

### 优点

- **逐字节可复现**：sha256 种子、无 `random` 无时钟，`(seed,dirtiness,link,domain)` 相同即产物完全一致（`test_same_knobs_byte_identical`）；天然适合基准、快照测试、缓存与 diff。
- **真值先建、始终可恢复**：oracle 恒 1.0；两旋钮只动观测、永不改真值（多项测试全组合验证）；污染全部记入 `corruption_map`（variant→canonical），可逆。
- **判别力有"牙齿"**：link 隐晦或脏度升高时 naive 崩、linked 仍能部分恢复，gap 由负转正（L1 `gap=-0.200` → L≥2 `gap=+0.800`），客观证明任务需要跨源推理。
- **核心零依赖、纯 stdlib**：仅 `hashlib/json/csv/sqlite3/argparse/math`，`pyproject` `dependencies=[]`；部署与审计成本极低。UI 依赖被隔离为可选项。
- **spec 驱动、换域零代码**：pattern 与 domain 词表/角色分离，下游全部 role-generic；新域只写一个 spec JSON。
- **多模态真实落地**：JSON/CSV/**可查询 SQLite**/人读 preview 四种；SQLite 是真正关系型、能跑 SQL（`test_sqlite_is_queryable` 验证）。
- **CLI 与 Web UI 同一条生成路径**：无第二套代码，UI 展示与 CLI/库产物一致，降低"演示与实测不符"的风险。
- **测试覆盖到位**：6 个测试文件、29 个用例，覆盖确定性、真值恢复、脏度可逆、三解法判别区间、落地模态、CLI 连线。

### 缺点 / 局限（如实标注）

- **分布是手设合成示意，未逆向自真实数据（no real-data calibration）**：`vocab` 里的 measure_range、metric range、`_wiggle` 频率/幅度、各污染阈值都是手工设定的合成值，不是从真实物流数据校准而来。spec 的 `notes.honesty` 与 generator/generator docstring 均明确标注。不能把它当"真实世界分布"用。
- **linked 是确定性占位替身，不是真正的语义求解器**：它只区分 **L1（显式 key）vs L≥2（非显式）**，L2–L5 得分相同（约 0.8），无法体现 L2→L5（尤其纯语义 L5）的细粒度梯度。那条梯度是留给**未来 LLM/axiom 解法**去跑的，本包只提供确定性替身。
- **linked 满分上限 ~0.8，不是 1.0**：这是骨架解法的天花板，因此 L1 下会出现 `gap` 为负（naive 1.0 > linked 0.8）—— 这是设计使然，不是 bug，但读者需知它并非"越跨源越高分"。
- **鲁棒性下降是跨 seed 均值趋势、非逐 seed 单调**：`test_dirtiness_degrades_linked_on_the_mean` 明确注释——单个 seed 上污染偶尔会把边界匹配"歪打正着"，只有对 8 个 seed 取均值后才是近似单调曲线。不要据单个 seed 下结论。
- **无 PDF/NoSQL 多模态、无 LLM 基准跑**：本包只有 SQL + timeseries + news 三种 store，以及确定性三解法。真正的 agentic 解析器（§5）、PDF/NoSQL 模态（§3）、RESEARCH 的 LLM-ablation 基准是母项目 Prism 的另一条线，**不在本包内**（spec `notes.scope` 已列明"未做"清单）。
- **numeric 冻结污染不进 corruption_map**：其余 5 类污染都有 variant→canonical 记录、可逆；数值冻结直接改写吞吐序列、不单列记录，故这一类不可从 map 精确还原。
- **判别力解法仅覆盖 `explain_delays`**：`naive_solve`/`linked_solve` 对非 `explain_delays` 任务直接返回空；`anomaly_cause` 目前只有 oracle 侧答案，没有对应的 naive/linked 实现。
- **单进程、单机、内存内**：无并发/分布式/流式，规模由 spec counts 决定（示例为个位数/几十量级）；适合基准示例，不是大数据管线。
- **clean-room 抽取**：从个人学习沙盒中抽出，仅含本项目自有代码；`_unit`/`_wiggle` 等原语是"verbatim from the Prism sandbox"，与母项目共享设计但独立成包。
